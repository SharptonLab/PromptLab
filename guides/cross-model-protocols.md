# Cross-Model Validation Protocols

## Purpose

Protocols for using multiple LLMs to validate outputs, leveraging the fact that different models have different training data, failure modes, and biases. This document describes workflows, not single prompts.

## Core Principle

Different LLMs make different mistakes. When two or more models agree on a factual claim, confidence increases. When they disagree, you've identified something requiring independent verification.

**Important caveat**: Agreement between models does not prove correctness. Models can share training data and reproduce the same errors. Cross-model validation supplements human verification; it does not replace it.

---

## When to Use Cross-Model Validation

**Good use cases:**
- Factual claims where you need extra confidence
- Statistical recommendations where you want to check consensus
- Code generation where logic errors might slip through
- Identifying claims that warrant external verification

**Poor use cases:**
- Creative writing (different ≠ wrong)
- Subjective judgments (no ground truth)
- Tasks where you'll verify independently anyway
- Simple queries where overhead exceeds value

## Protocol 1: Independent Generation + Comparison

### Purpose
Generate answers from multiple models independently, then compare to identify agreement, disagreement, and gaps.

### Workflow

**Step 1: Get independent responses**

Submit the same prompt to 2-3 models. Do not share previous responses.

```
[Identical prompt to Model A, Model B, Model C]
```

**Step 2: Compare responses**

Create a comparison table:

| Aspect | Model A | Model B | Model C | Agreement? |
|--------|---------|---------|---------|------------|
| Main claim | | | | |
| Key details | | | | |
| Citations | | | | |
| Recommendations | | | | |

**Step 3: Interpret disagreements**

For each disagreement:
1. Is one model clearly wrong (verifiable error)?
2. Are both valid alternatives?
3. Is this an area of genuine uncertainty?

**Step 4: Verify contested claims**

Claims where models disagree are high-priority for external verification.

### Example

**Prompt to each model:**
"What is the recommended sample size for a 16S rRNA microbiome study comparing two groups, assuming you want 80% power to detect a moderate effect?"

**Model A response:** "20-30 samples per group based on simulation studies (Kelly et al., 2015)"

**Model B response:** "At least 20 samples per group is common guidance, though this varies by effect size. La Rosa et al. (2012) specifically examined microbiome power."

**Model C response:** "Generally 15-30 per group, depending on expected effect size and outcome variable."

**Analysis:**
- Agreement: 20-30 range is reasonable
- Disagreement: Different citations (Kelly vs. La Rosa)
- Action: Verify both citations; the range is probably reasonable but citations need checking

## Protocol 2: Generation + Adversarial Critique

### Purpose
Use one model to generate content, another to critique it, leveraging the observation that models are less sycophantic about other models' outputs.

### Workflow

**Step 1: Generate with Model A**
```
[Your task prompt]
```

**Step 2: Critique with Model B**
```
The following text was generated by another AI system. Please critically evaluate it for errors, weaknesses, unsupported claims, and potential hallucinations. Be specifically critical of any citations, statistics, or factual claims.

[Paste Model A output]
```

**Step 3: Optional - have Model A respond to critique**
```
A reviewer has raised the following concerns about your previous response. Please address each concern:

[Paste Model B critique]
```

### Why This Works

Models trained to be helpful are less defensive about output they didn't generate. The critique may be harsher and more genuine than self-critique.

## Protocol 3: Citation Cross-Check

### Purpose
Specifically validate citations using multiple models before manual verification.

### Workflow

**Step 1: Extract citations from output**

After getting LLM output with citations, list all citations.

**Step 2: Cross-check each citation**

For each citation, ask a different model:
```
Can you provide details about this paper: [citation]

Specifically:
1. Do you recognize this paper?
2. What journal was it published in?
3. What were its main findings?
4. Is there anything unusual about this citation?

If you're not sure whether this paper exists, say so explicitly.
```

**Step 3: Interpret responses**

| Citation | Model A generated | Model B recognizes? | Consistent details? | Likely status |
|----------|------------------|---------------------|---------------------|---------------|
| | | | | Real / Uncertain / Likely fabricated |

**Step 4: Verify in databases**

Check PubMed/Google Scholar for any citation flagged as uncertain.

### Caveat

Both models may "recognize" fabricated citations by generating consistent-seeming details. This protocol flags likely fabrications but doesn't prove citations are real.

## Protocol 4: Code Validation

### Purpose
Use multiple models to review generated code for logic errors, bugs, and issues.

### Workflow

**Step 1: Generate code with Model A**

**Step 2: Review with Model B**
````
Please review this code for correctness, potential bugs, and logic errors. Do not assume it is correct.

```{language}
[paste code]
```

Specifically check:
1. Does the logic match the stated purpose?
2. Are there edge cases that would cause errors?
3. Are there any obvious bugs?
4. Is the approach appropriate for the task?
```

**Step 3: Compare implementations**

Optionally, ask Model B to write its own implementation:
```
Write code to accomplish [same task]. Do not look at the previous implementation—write your own approach.
````

Compare the two implementations. If they differ significantly:
- Understand why
- Test both
- One may reveal bugs in the other

## Protocol 5: Consensus Building for Recommendations

### Purpose
Gather recommendations from multiple models to identify consensus and contested areas.

### Workflow

**Step 1: Get recommendations from each model**
```
[Same question to Model A, Model B, Model C]
```

**Step 2: Synthesize with explicit comparison**

Ask one model to synthesize:
```
I asked three different AI systems for recommendations on [topic]. Here are their responses:

Model A: [response]
Model B: [response]
Model C: [response]

Please analyze:
1. What do they agree on? (Likely more reliable)
2. What do they disagree on? (Requires verification)
3. What does each suggest that the others don't mention?
4. What's the most defensible recommendation given this input?
```

## Interpreting Cross-Model Results

### Strong agreement
All models give consistent answers with consistent details.
- **Interpretation**: Higher confidence, but not proof
- **Action**: Reduced verification priority (but still verify high-stakes claims)

### Partial agreement
Models agree on main points but differ on details.
- **Interpretation**: Main point more reliable; details need checking
- **Action**: Verify the differing details specifically

### Disagreement
Models give conflicting answers.
- **Interpretation**: At least one is wrong; possibly all are wrong
- **Action**: Prioritize for external verification

### One confident, others uncertain
One model makes strong claims; others hedge or admit uncertainty.
- **Interpretation**: The confident model may be hallucinating
- **Action**: Verify the strong claims; consider the hedging informative

## Practical Considerations

### Model Selection

For effective cross-validation, use models from different providers/architectures:
- Claude (Anthropic)
- GPT-4 (OpenAI)
- Gemini (Google)
- Open models (Llama, Mistral)

Using multiple versions of the same model family (e.g., Claude 3 and Claude 3.5) provides less independent validation.

### Cost and Time

Cross-model validation adds overhead:
- 2-3x API calls or subscriptions
- Time to compare outputs
- Cognitive effort to synthesize

Reserve for high-stakes outputs where validation overhead is justified.

### Documentation

Record which models you used and what you found:
```
Cross-model validation performed:
- Primary generation: Claude 3.5
- Validation: GPT-4
- Key disagreement: [describe]
- Resolution: [describe what you did]
```

---

## Limitations

1. **Shared training data**: Models may reproduce the same false information if it's common in training data
2. **Correlated errors**: Models may make similar types of errors (e.g., both fabricate plausible citations)
3. **False confidence from agreement**: Agreement increases confidence but doesn't guarantee truth
4. **Subjective disagreement**: Some disagreements reflect legitimate differences, not errors
5. **Overhead**: Cross-validation isn't free—balance effort against value

## Quick Reference: When to Use Which Protocol

| Scenario | Recommended Protocol |
|----------|---------------------|
| Factual claims to verify | Protocol 1: Independent Generation |
| Suspect hallucinations in output | Protocol 2: Adversarial Critique |
| Citations to validate | Protocol 3: Citation Cross-Check |
| Generated code to review | Protocol 4: Code Validation |
| Need recommendation on approach | Protocol 5: Consensus Building |
