### 3. Manual Validation Checks (For Real Data)
After passing automated tests, inspect your actual research data with these quick checks:
1. **Per-sample mean check**: For every sample (row) in your output, compute the mean of CLR values. It will be *extremely close to 0* (absolute difference < 1e-10, due to floating-point precision). This is the non-negotiable defining property of CLR.
2. **Relative abundance preservation**: For a given sample, if Taxon A has a higher raw count than Taxon B, its CLR value must also be higher. CLR preserves the order of abundances within a sample.
3. **Value range check**: For typical microbiome data (counts 0–1e5), CLR values usually fall between -15 and 15. Values outside this range may indicate data errors or incorrect transformation.
4. **Quick scaling test**: Take one sample, multiply all its counts by 2, re-run CLR, and confirm the output is identical to the original. This validates per-sample normalization.

### 4. Red Flags (Indicate Code Errors)
These output patterns almost always signal bugs:
| Red Flag | Likely Cause |
|----------|--------------|
| Per-sample (row) mean is not ~0 | Used arithmetic mean instead of geometric mean, or used `axis=0` instead of `axis=1` in the mean calculation (centers per feature instead of per sample) |
| Output contains `NaN`/`Inf` | Missing pseudocount (leads to `log(0) = -inf`) or division by zero (geometric mean = 0, impossible with a positive pseudocount) |

| Scaling a sample’s counts changes its CLR values | Using a global geometric mean (across all samples) instead of per-sample geometric mean |
| All CLR values for a feature are identical across samples | Same as above: global normalization instead of per-sample |
