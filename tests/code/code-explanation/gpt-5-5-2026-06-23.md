## 3. Key concepts

### DESeq2 models counts, not relative abundances

DESeq2 was originally developed for RNA-seq count data, but it is sometimes used in microbiome differential abundance analysis. It expects raw integer counts.

Microbiome data have special properties, especially compositionality, meaning that counts are constrained by sequencing depth and represent relative sampling from a community. This makes microbiome differential abundance analysis more complicated than ordinary count analysis.

DESeq2 can be used, but results should be interpreted carefully.

---

### The design formula

The formula:

```r
~ treatment + batch
```

means:

> Include treatment and batch as explanatory variables.

It does not mean treatment divided by batch or treatment nested in batch. It means both variables are included in the model.

This is equivalent to saying:

```r
counts ~ treatment + batch
```

but in DESeq2 the response variable, the count matrix, is already known, so the formula only contains the right-hand side.

If you had written:

```r
design = ~ treatment
```

then DESeq2 would ignore batch.

If you had written:

```r
design = ~ batch + treatment
```

that would usually give the same treatment comparison. In DESeq2, people often put the main variable of interest last, but with explicit `contrast`, the order is less important.

---

### Contrast

The contrast tells DESeq2 which group comparison you want.

```r
contrast = c("treatment", "diseased", "healthy")
```

means:

```r
diseased - healthy
```

on the log2 scale.

So the reported fold change is:

```r
log2(diseased / healthy)
```

If you reversed the order:

```r
contrast = c("treatment", "healthy", "diseased")
```

then all log2 fold changes would change sign.

---

### Log2 fold change

A log2 fold change describes the size of the difference between groups.

| log2FoldChange | Fold change | Interpretation |
|---:|---:|---|
| 0 | 1 | no difference |
| 1 | 2 | 2x higher in diseased |
| 2 | 4 | 4x higher in diseased |
| -1 | 0.5 | 2x lower in diseased |
| -2 | 0.25 | 4x lower in diseased |

So the threshold:

```r
abs(log2FoldChange) > 1
```

keeps taxa with more than a 2-fold difference in either direction.

---

### Adjusted p-values

DESeq2 tests many taxa. If you test hundreds or thousands of taxa, some will appear significant just by chance.

The `padj` column contains p-values adjusted for multiple testing, usually using the Benjamini-Hochberg false discovery rate method.

A common cutoff is:

```r
padj < 0.05
```

meaning that among the features called significant, the expected false discovery rate is controlled around 5%.

---

### Why use both `lfcThreshold = 1` and `abs(log2FoldChange) > 1`?

This is a very good question.

The line:

```r
results(dds, ..., lfcThreshold = 1)
```

changes the statistical test so that DESeq2 tests for evidence of a fold change larger than the threshold.

So the adjusted p-value `padj` is not just asking:

> Is log2 fold change different from 0?

It is asking something stricter, approximately:

> Is the absolute log2 fold change significantly greater than 1?

Then the later filter:

```r
abs(res$log2FoldChange) > 1
```

also requires the estimated effect size itself to be larger than 1.

This can seem redundant, but it is often used to make the final table explicit:

```r
res_sig <- taxa with significant padj and estimated |log2FC| > 1
```

However, there is some duplication in logic.

A cleaner version could be:

```r
res_sig <- res[which(res$padj < 0.05), ]
```

because the `padj` values were already computed using `lfcThreshold = 1`.

Or, if not using `lfcThreshold`, one might write:

```r
res <- results(dds, contrast = c("treatment", "diseased", "healthy"),
               alpha = 0.05)

res_sig <- res[which(res$padj < 0.05 & abs(res$log2FoldChange) > 1), ]
```

That version tests for any nonzero difference, then filters for effect size afterward.

The published code combines both approaches: it uses a threshold-aware statistical test and then also filters by the estimated log2 fold change.

That is conservative, but worth noting.

## 5. Potential issues and caveats

### 1. Microbiome data are compositional

DESeq2 is widely used, but microbiome count data are compositional: the total sequencing depth constrains observed counts. An increase in one taxon can make others appear to decrease, even if their absolute abundances did not change.

For microbiome-specific differential abundance, other methods are also commonly used, such as:

- ANCOM-BC
- ALDEx2
- MaAsLin2
- corncob
- Songbird
- LinDA

That does not mean this code is wrong, but results should be interpreted with microbiome-specific caution.

---

### 2. `otu_counts` must be raw integer counts

This code is not appropriate if `otu_counts` contains:

- relative abundances
- percentages
- rarefied counts
- centered log-ratio transformed values
- normalized counts from another method

DESeq2 wants raw counts.

---

### 3. The filtering rule is simple and somewhat arbitrary

```r
rowSums(counts(dds)) >= 10
```

keeps taxa with at least 10 total counts across all samples.

This may keep a taxon observed only in one sample with 10 reads, while removing a taxon observed in several samples with low counts.

For microbiome data, a prevalence filter may be more biologically reasonable, for example:

```r
keep <- rowSums(counts(dds) >= 5) >= 3
dds <- dds[keep, ]
```

meaning: keep taxa with at least 5 reads in at least 3 samples.

The best filter depends on the study design and sequencing depth.

---

### 4. Treatment and batch must not be perfectly confounded

The model:

```r
~ treatment + batch
```

only works if DESeq2 can distinguish treatment effects from batch effects.

If all healthy samples are in batch A and all diseased samples are in batch B, then treatment and batch are perfectly confounded. In that case, DESeq2 cannot know whether differences are due to disease or batch.

You can check with:

```r
table(sample_metadata$treatment, sample_metadata$batch)
```

A balanced design is much better, for example:

|        | batch A | batch B |
|--------|---------|---------|
| healthy | 10 | 10 |
| diseased | 10 | 10 |

---

### 5. `batch` should usually be a factor

If `batch` is coded as `1`, `2`, `3`, R may treat it as numeric unless converted.

That would model batch as a continuous trend, which is usually not what you want.

Better:

```r
sample_metadata$batch <- factor(sample_metadata$batch)
```

Similarly:

```r
sample_metadata$treatment <- factor(sample_metadata$treatment)
```

---

### 6. Missing `padj` values can affect the final filter

Some rows in `res$padj` may be `NA`, often due to very low counts or independent filtering.

This line:

```r
res_sig <- res[which(res$padj < 0.05 & abs(res$log2FoldChange) > 1), ]
```

will generally exclude `NA` adjusted p-values, because `which()` ignores `NA` values.

A more explicit version would be:

```r
res_sig <- res[!is.na(res$padj) &
               res$padj < 0.05 &
               abs(res$log2FoldChange) > 1, ]
```

That is clearer and safer.

---

### 7. Log2 fold changes may benefit from shrinkage

DESeq2 log2 fold-change estimates for low-count taxa can be noisy. DESeq2 often recommends using log fold-change shrinkage for visualization and ranking.

For example:

```r
res_shrunk <- lfcShrink(dds,
                        contrast = c("treatment", "diseased", "healthy"),
                        type = "ashr")
```

However, the exact use depends on the DESeq2 version and whether you want shrunken estimates for ranking, plotting, or reporting.

---

### 8. No interaction between treatment and batch is modeled

The design:

```r
~ treatment + batch
```

assumes the treatment effect is the same across batches.

If you believe the treatment effect differs by batch, you would need an interaction model:

```r
~ batch + treatment + batch:treatment
```

or equivalently:

```r
~ batch * treatment
```

But this requires enough samples in each batch-treatment combination.
