````
### B. The design formula: `~ treatment + batch`

```r
design = ~ treatment + batch
```

#### What it means

This tells DESeq2 to model each taxon’s counts as a function of both:

1. `treatment`
2. `batch`

In words:

> For each taxon, estimate the effect of treatment while adjusting for batch.

This is important if samples were processed in different sequencing runs, extraction batches, centers, plates, etc.

#### Why include `batch`?

Suppose diseased samples were mostly sequenced in batch 1 and healthy samples mostly in batch 2. Then apparent “disease-associated” taxa might actually be “batch-associated” taxa.

By including `batch`, DESeq2 tries to separate:

- variation explained by disease status
- variation explained by batch

So the treatment comparison becomes:

> diseased versus healthy, controlling for batch differences.

#### Important detail: additive model

The formula:

```r
~ treatment + batch
```

means DESeq2 assumes the treatment effect is the same across batches.

It does **not** model an interaction between treatment and batch. If you expected the disease effect to differ by batch, you might need something like:

```r
~ batch + treatment + batch:treatment
```

or equivalently:

```r
~ batch * treatment
```

But that requires more careful interpretation.

#### Order of terms

In this specific code, because the results are later extracted using an explicit `contrast`, the order of `treatment` and `batch` is not very important.

However, in DESeq2, if you simply ran:

```r
results(dds)
```

without specifying a contrast, DESeq2 would usually return the result for the last term in the design. Here they avoid ambiguity by explicitly requesting the treatment comparison.

### C. Filter low-count taxa/features

```r
dds <- dds[rowSums(counts(dds)) >= 10, ]
```

#### What it does

This removes taxa whose total count across all samples is less than 10.

Breaking it down:

```r
counts(dds)
```

extracts the count matrix from the DESeq2 object.

```r
rowSums(counts(dds))
```

calculates the total count for each taxon across all samples.

```r
rowSums(counts(dds)) >= 10
```

creates a logical vector saying whether each taxon has at least 10 total counts.

Then:

```r
dds[ ..., ]
```

keeps only those taxa.

#### Example

If a taxon has counts:

```r
0, 0, 1, 0, 3, 0
```

its total count is 4, so it would be removed.

If another taxon has:

```r
0, 5, 2, 1, 0, 4
```

its total count is 12, so it would be kept.

#### Why filter low-count taxa?

Very rare taxa have several problems:

1. **Little statistical information**  
   If a taxon appears only a few times total, it is hard to estimate differences reliably.

2. **Unstable fold changes**  
   Going from 0 counts to 3 counts can look like a huge fold change, but may not be biologically meaningful.

3. **Multiple testing burden**  
   DESeq2 tests each taxon separately. Removing mostly uninformative taxa reduces the number of tests, which can improve power.

4. **Computational efficiency**  
   Fewer features means faster analysis.

#### Is `>= 10` a universal cutoff?

No. It is a simple rule of thumb. In microbiome studies, people may use stronger filters, such as keeping taxa present above some count threshold in a minimum number of samples.

For example:

```r
keep <- rowSums(counts(dds) >= 5) >= 3
dds <- dds[keep, ]
```

That would keep taxa with at least 5 counts in at least 3 samples.

The best filter depends on study size, sequencing depth, sparsity, and biological goals.

### D. Run DESeq2

```r
dds <- DESeq(dds)
```

#### What it does

This is the main DESeq2 analysis step.

Internally, DESeq2 does several things:

1. **Estimate size factors**  
   These normalize for differences in sequencing depth/library size between samples.

2. **Estimate dispersion**  
   Dispersion describes how variable the counts are beyond simple Poisson counting noise. DESeq2 uses a negative binomial model, which is common for overdispersed count data.

3. **Fit a model for each taxon**  
   For each taxon, DESeq2 fits a model using the design formula:

   ```r
   ~ treatment + batch
   ```

4. **Perform statistical tests**  
   By default, DESeq2 uses a Wald test to test coefficients in the model.

#### Important microbiome note

DESeq2 was originally developed for RNA-seq count data. It is often used for microbiome count data, but microbiome data have additional complications:

- compositionality
- many zeros
- uneven sparsity across taxa
- strong sample-to-sample variation

So DESeq2 can be useful, but its assumptions should be considered carefully.

contrast = c("variable_name", "numerator_level", "denominator_level")

If a taxon has:

```r
log2FoldChange = 2

```r
2^2 = 4

If:

```r

2^1 = 2

| log2FoldChange | Interpretation |
|---:|---|
| 2 | 4 times higher in diseased |
| 1 | 2 times higher in diseased |

## The `alpha` argument

```r
alpha = 0.05
```

This tells DESeq2 the target significance level for adjusted p-values, mainly used for DESeq2’s independent filtering procedure.

It does **not** automatically remove non-significant taxa from `res`.

The result object `res` will still contain all tested taxa, including non-significant ones. You later manually filter with:

```r
res$padj < 0.05
```

## The `lfcThreshold` argument

```r
lfcThreshold = 1
```

This is an important detail.

`lfcThreshold = 1` means DESeq2 tests whether the absolute log2 fold change is greater than 1.

Because log2 fold change of 1 corresponds to a 2-fold difference:

```r
2^1 = 2
```

this asks:

> Is this taxon significantly more than 2-fold different between diseased and healthy?

By default, when you use:

```r
lfcThreshold = 1
```

DESeq2 tests against the null hypothesis:

```r
abs(log2FoldChange) <= 1
```

rather than the usual null hypothesis:

```r
log2FoldChange = 0
```

So instead of asking:

> Is there any difference at all?

it asks:

> Is the difference large enough to exceed a 2-fold change?

This is often useful because tiny differences can be statistically significant in large datasets but may not be biologically meaningful.

### F. Filter significant results

```r

#### What it does

This keeps taxa that satisfy both:

```r
res$padj < 0.05
```

and

```r
abs(res$log2FoldChange) > 1
```

So the final result contains taxa with:

1. adjusted p-value less than 0.05
2. absolute log2 fold change greater than 1

In biological terms:

> taxa that are statistically significant after multiple-testing correction and have more than a 2-fold difference between diseased and healthy.

---

## Why use `padj`?

`padj` means adjusted p-value.

Because DESeq2 tests many taxa, raw p-values need correction for multiple testing. If you test 1,000 taxa, some will appear significant just by chance.

DESeq2 usually uses the Benjamini-Hochberg false discovery rate procedure.

So:

```r
padj < 0.05
```

roughly means the analysis is controlling the false discovery rate at 5%.

---

## Why use `abs(res$log2FoldChange) > 1`?

The function `abs()` takes the absolute value.

So this condition keeps both:

```r
log2FoldChange > 1
```

and

```r
log2FoldChange < -1
```

That means it keeps taxa that are either:

- more abundant in diseased by more than 2-fold
- less abundant in diseased by more than 2-fold

Without `abs()`, you would only get taxa higher in diseased.

---

## Why check both `padj` and `log2FoldChange` when `lfcThreshold` was already set?

This is a subtle but important point.

Because they used:

```r
lfcThreshold = 1
```

the `padj` values are already based on testing whether the effect is larger than a log2 fold change of 1.

So in many cases this final check:

```r
abs(res$log2FoldChange) > 1
```

is partly redundant.

The statistically important part is:

```r
res$padj < 0.05
```

because with `lfcThreshold = 1`, that adjusted p-value already corresponds to a test of whether the effect exceeds the threshold.

However, people often include both conditions for clarity or extra caution:

```r
res$padj < 0.05 & abs(res$log2FoldChange) > 1
```

It makes explicit that the final table contains taxa with both statistical significance and observed effect size above the threshold.

A cleaner version would be:

```r
res_sig <- res[!is.na(res$padj) &
               res$padj < 0.05 &
               abs(res$log2FoldChange) > 1, ]
```

But conceptually, with `lfcThreshold = 1`, the extra fold-change filter is not usually necessary.

### Formula syntax in R

The formula:

```r

```

means:

> model the response using treatment and batch as predictors.

In DESeq2, the response is the count for each taxon.

A simple formula:

```r
~ treatment
```

would compare treatment groups without adjusting for anything else.

A formula with batch:

```r
~ treatment + batch
```

compares treatment groups while adjusting for batch.

### Contrast syntax in DESeq2

This:

```r

```

means:

```r

```

on the log2 scale.

Positive log2 fold changes mean higher in diseased.

Negative log2 fold changes mean higher in healthy.

### Log2 fold change

DESeq2 reports fold changes on the log2 scale.

Useful conversions:

```r
log2FC = 1   means 2-fold higher
log2FC = 2   means 4-fold higher
log2FC = -1  means 2-fold lower
log2FC = -2  means 4-fold lower
```

You can convert log2 fold change to ordinary fold change using:

```r
2^res$log2FoldChange
```

### Adjusted p-values

`pvalue` is the raw p-value.

`padj` is the multiple-testing-adjusted p-value.

In high-dimensional data like microbiome data, you usually interpret `padj`, not `pvalue`.

## 5. Potential issues and limitations

### 1. DESeq2 expects raw counts

`otu_counts` should be raw integer counts.

It should not be:

- relative abundance
- proportions
- percentages
- log-transformed values
- CSS-normalized values
- TPM-like normalized values

DESeq2 estimates its own normalization factors.

---

### 2. Microbiome data are compositional

Microbiome sequencing data are compositional: an increase in one taxon can make others appear relatively decreased because the sequencing depth is constrained.

DESeq2 models counts and adjusts for library size, but it does not fully solve compositionality. Depending on the study question, alternatives such as ANCOM-BC, ALDEx2, MaAsLin2, corncob, or other compositional methods may be considered.

---

### 3. Many zeros can cause problems

Microbiome count tables often contain many zeros. DESeq2’s default size-factor estimation can have trouble when every taxon has at least one zero across samples.

For microbiome data, people sometimes use:

```r
dds <- estimateSizeFactors(dds, type = "poscounts")
```

or pass this through the `DESeq()` workflow with care.

The given code uses the default normalization unless otherwise set.

---

### 4. Batch and treatment must not be perfectly confounded

The design:

```r
~ treatment + batch
```

only works if there is enough information to separate treatment effects from batch effects.

For example, this is problematic:

| sample | treatment | batch |
|---|---|---|
| sample1 | healthy | batch1 |
| sample2 | healthy | batch1 |
| sample3 | diseased | batch2 |
| sample4 | diseased | batch2 |

Here, all healthy samples are in batch 1 and all diseased samples are in batch 2. DESeq2 cannot tell whether differences are due to treatment or batch.

You would get a design matrix problem or uninterpretable results.

---

### 5. `batch` should usually be a factor

If `batch` is categorical, make sure it is encoded as a factor:

```r
sample_metadata$batch <- factor(sample_metadata$batch)
```

Similarly:

```r
sample_metadata$treatment <- factor(sample_metadata$treatment)
```

If `batch` is numeric, DESeq2 may treat it as a continuous variable, which may not be intended.

---

### 6. The filtering threshold is arbitrary

This filter:

```r
rowSums(counts(dds)) >= 10
```

is minimal and may leave in many sparse taxa.

Depending on the dataset, a prevalence-based filter may be more appropriate, such as:

```r
keep <- rowSums(counts(dds) >= 10) >= 5
dds <- dds[keep, ]
```

This keeps taxa with at least 10 counts in at least 5 samples.

---

### 7. `lfcThreshold` plus fold-change filtering is partly redundant

Because the results were generated using:

```r
lfcThreshold = 1
```

the adjusted p-values already relate to whether the effect is significantly larger than a log2 fold change of 1.

Therefore this:

```r
abs(res$log2FoldChange) > 1
```

is mostly an extra explicit filter. It is not necessarily wrong, but it should be understood as a conservative or clarifying step.

---

### 8. Log2 fold changes may be noisy for rare taxa

Even after filtering, rare taxa can have unstable fold-change estimates. DESeq2 offers log fold-change shrinkage methods, such as:

```r
res_shrunk <- lfcShrink(dds, contrast = c("treatment", "diseased", "healthy"),
                        type = "ashr")
```

Shrinkage is often useful for ranking taxa by effect size or making plots, though hypothesis testing with `lfcThreshold` has its own interpretation.

---

### 9. `padj` can be `NA`

Some taxa may have:

```r
padj = NA
```

This can happen due to low counts, outliers, or independent filtering.

The original code:

```r
res_sig <- res[which(res$padj < 0.05 & abs(res$log2FoldChange) > 1), ]
```

will usually exclude those because `which()` ignores `NA`, but a clearer version is:

```r
res_sig <- res[!is.na(res$padj) &
               res$padj < 0.05 &
               abs(res$log2FoldChange) > 1, ]
```
````
