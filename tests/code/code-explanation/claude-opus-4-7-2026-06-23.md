## 1. Overview

This code performs differential abundance analysis on microbiome OTU count data using DESeq2. It compares the abundance of microbial taxa between "diseased" and "healthy" samples while adjusting for batch effects, then filters the results to retain only statistically significant taxa with substantial fold changes.

## 2. Step-by-Step Walkthrough

### Section 1: Building the DESeq2 data object

```r
dds <- DESeqDataSetFromMatrix(countData = otu_counts,
                               colData = sample_metadata,
                               design = ~ treatment + batch)
```

This constructs a `DESeqDataSet` (a specialized container) by combining three inputs:
- **`countData`**: Your OTU table — rows are OTUs/taxa, columns are samples, values are raw integer counts.
- **`colData`**: Sample metadata — rows are samples (in the **same order** as columns in `countData`), columns describe each sample (treatment group, batch, etc.).
- **`design`**: A formula telling DESeq2 which variables to model. More on this below.

### Section 2: Low-count filtering

```r
dds <- dds[rowSums(counts(dds)) >= 10, ]
```

This keeps only OTUs whose total count across **all samples combined** is at least 10. OTUs with very few reads (e.g., singletons that appear in only one sample once) carry almost no statistical information, inflate multiple-testing corrections, and slow down computation. The threshold of 10 is a common but somewhat arbitrary choice — you'll see 5, 10, or "present in ≥X% of samples" in different papers.

### Section 3: Running the analysis

```r
dds <- DESeq(dds)
```

This single function performs three steps internally:
1. **Size factor estimation** — normalizes for differences in sequencing depth between samples.
2. **Dispersion estimation** — estimates how much each OTU's counts vary beyond what's expected by chance (microbiome data is "overdispersed").
3. **Negative binomial model fitting** — fits a statistical model for each OTU using the design formula.

### Section 4: Extracting results for a specific comparison

```r
res <- results(dds, contrast = c("treatment", "diseased", "healthy"),
               alpha = 0.05, lfcThreshold = 1)
```

- **`contrast = c("treatment", "diseased", "healthy")`**: Compares "diseased" vs "healthy" within the `treatment` column. The format is `c(variable_name, numerator, denominator)`. A positive log2 fold change means the OTU is **more abundant in diseased** (because diseased is the numerator).
- **`alpha = 0.05`**: The significance threshold used internally for independent filtering (an optimization that improves statistical power).
- **`lfcThreshold = 1`**: Tests whether the log2 fold change is significantly **greater than 1** in absolute value — i.e., the OTU must change by more than 2-fold to be considered significant. This is stricter than the default test (which only asks whether the fold change differs from zero).

### Section 5: Final filtering

```r

Retains rows where the adjusted p-value is below 0.05 **and** the absolute log2 fold change exceeds 1. The `which()` wrapper safely handles `NA` values (DESeq2 sets `padj` to `NA` for OTUs that fail independent filtering).

## 3. Key Concepts (Answers to Your Questions)

### What does `~ treatment + batch` mean?

This is an **R formula** specifying a linear model. Read it as: "model the OTU counts as a function of treatment **while accounting for** batch."

- The **last variable** (`treatment` here would be more typical to put last, but DESeq2 specifically looks at the last variable as the "variable of interest" by default).
- `batch` is included as a **covariate** — DESeq2 estimates and removes its effect so that what remains is the cleaner treatment effect.

⚠️ **Note**: Actually, in DESeq2, the **last** variable in the design is treated as the variable of interest by default. So `~ batch + treatment` is the more conventional ordering. Since this code uses `contrast = c("treatment", ...)` explicitly, the ordering doesn't actually affect the result here — but it's a subtle convention worth knowing.

### Why filter `rowSums >= 10`?

Three reasons:
1. **Statistical**: Very low-count OTUs have unreliable dispersion estimates.
2. **Multiple testing**: Fewer tests means less aggressive p-value correction (FDR), so real signals are easier to detect.
3. **Speed**: Fewer rows = faster computation.

### What does `contrast = c("treatment", "diseased", "healthy")` do?

It specifies *which* pairwise comparison you want from your model. If `treatment` has multiple levels (e.g., healthy, mild, severe), DESeq2 needs to know which two groups to compare and in which direction. The syntax is:

```
c("column_name_in_metadata", "level_to_be_numerator", "level_to_be_denominator")
```

Positive log2FoldChange → higher in numerator (diseased).
Negative log2FoldChange → higher in denominator (healthy).

### What is `lfcThreshold` and why use it?

By default, DESeq2 tests the null hypothesis: "log2 fold change = 0" (i.e., any change at all is detected). With biological count data and large sample sizes, you can detect tiny, biologically meaningless changes as "statistically significant."

`lfcThreshold = 1` changes the null hypothesis to: "|log2 fold change| ≤ 1" — i.e., it formally tests whether the effect is **larger than 2-fold**. This produces **more conservative p-values** that already reflect a biological effect size threshold.

### Why check `padj` AND `log2FoldChange` again?

Good catch — this is partially redundant but not entirely:

- When `lfcThreshold = 1` is set, the p-values already incorporate this threshold, so OTUs with `padj < 0.05` *should* generally have |LFC| > 1.
- **However**, due to shrinkage and the way the test statistic is constructed, there can be edge cases where a significant `padj` corresponds to an estimated LFC slightly below 1.
- The explicit `abs(res$log2FoldChange) > 1` filter ensures **none** slip through.

It's a "belt and suspenders" approach — slightly redundant but safe. Some would argue it's unnecessary given `lfcThreshold` was already set.

## 5. Potential Issues and Limitations

1. **Microbiome-specific concerns**: DESeq2 was designed for RNA-seq, not microbiome data. Microbiome counts are often more sparse (lots of zeros) and **compositional** (relative, not absolute). Many microbiome researchers prefer tools like **ANCOM-BC**, **ALDEx2**, or **corncob** for this reason. DESeq2 is still widely used but is increasingly debated for 16S/shotgun data.

4. **Batch confounding**: If `batch` and `treatment` are confounded (e.g., all diseased samples in batch 1, all healthy in batch 2), the model cannot separate them, and DESeq2 will warn or error.

5. **No LFC shrinkage applied**: Modern DESeq2 workflows often use `lfcShrink()` to produce more reliable fold change estimates, especially for low-count OTUs. This code does not.

6. **OTU vs ASV terminology**: The variable is called `otu_counts`, but the same workflow applies to ASVs (amplicon sequence variants) from DADA2.
