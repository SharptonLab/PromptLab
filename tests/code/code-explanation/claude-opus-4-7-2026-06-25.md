# DESeq2 Differential Abundance Analysis — Explained

## 1. Overview

This code performs **differential abundance analysis** on microbiome count data using DESeq2. It identifies which OTUs (Operational Taxonomic Units, i.e., microbial taxa) differ significantly in abundance between **diseased** and **healthy** samples, while statistically adjusting for **batch effects**. The output is a filtered table of OTUs with both statistically significant and biologically meaningful changes.

## 2. Step-by-step Walkthrough

### Step 1: Build the DESeq2 dataset object

```r
dds <- DESeqDataSetFromMatrix(countData = otu_counts,
                              colData = sample_metadata,
                              design = ~ treatment + batch)
```

- **`countData`**: A matrix of raw integer counts (rows = OTUs, columns = samples). DESeq2 requires *raw* counts, not normalized or transformed values.
- **`colData`**: A data frame whose rows correspond (in order!) to the columns of `countData`. It contains sample-level metadata like `treatment` and `batch`.
- **`design = ~ treatment + batch`**: This is an **R formula**. It tells DESeq2 the statistical model:
  - The `~` means "modeled as a function of."
  - `treatment + batch` means "estimate the effect of treatment *while controlling for* batch."
  - The **last variable** (`treatment` here would be, but actually it's the variable you want to test) is typically the variable of interest. Best practice: put the variable of interest **last**, not first. Here `batch` is listed last, which is unusual — you may want `~ batch + treatment` instead. *(Note: with the explicit `contrast` argument later, ordering matters less, but it's still a convention worth following.)*

### Step 2: Filter low-count OTUs

```r
dds <- dds[rowSums(counts(dds)) >= 10, ]
```

- `counts(dds)` extracts the count matrix.
- `rowSums(...)` sums each row (i.e., total counts per OTU across all samples).
- `>= 10` keeps OTUs with at least 10 total counts.
- The `[rows, ]` syntax subsets the dataset, keeping only those OTUs.

**Why?** OTUs with very few counts have almost no statistical power and just add noise plus inflate multiple-testing corrections. The threshold of 10 is a common rule of thumb but is somewhat arbitrary. Some workflows use stricter filters (e.g., present in ≥10% of samples).

### Step 3: Run the DESeq2 analysis

```r
dds <- DESeq(dds)
```

This single function does three things internally:
1. **Estimates size factors** (normalization for sequencing depth differences).
2. **Estimates dispersion** (how variable each OTU is across samples).
3. **Fits a Negative Binomial GLM** and tests coefficients via Wald tests.

### Step 4: Extract results for a specific comparison

```r
res <- results(dds, contrast = c("treatment", "diseased", "healthy"),
               alpha = 0.05, lfcThreshold = 1)
```

- **`contrast = c("treatment", "diseased", "healthy")`**: This specifies *which comparison* you want.
  - Format: `c(factor_name, numerator_level, denominator_level)`.
  - This computes log2(diseased / healthy). So **positive log2FoldChange = higher in diseased**, negative = higher in healthy.
- **`alpha = 0.05`**: The significance threshold used for independent filtering and reporting. It should match the cutoff you'll use downstream.
- **`lfcThreshold = 1`**: This is important and often misunderstood (see below).

#### What `lfcThreshold` does

By default, DESeq2 tests the null hypothesis: "log2FoldChange = 0" (i.e., any change at all).

With `lfcThreshold = 1`, the null hypothesis becomes: "|log2FoldChange| ≤ 1." DESeq2 now tests whether the fold change is **significantly greater than 2-fold** (since 2^1 = 2). This produces more conservative p-values that incorporate effect-size meaningfulness directly into the statistical test — not just statistical significance.

### Step 5: Filter to significant results

```r
res_sig <- res[which(res$padj < 0.05 & abs(res$log2FoldChange) > 1), ]
```

- `res$padj`: Benjamini-Hochberg adjusted p-values (false discovery rate, FDR).
- `abs(res$log2FoldChange) > 1`: Effect size of >2-fold in either direction.
- `which(...)`: Returns indices of `TRUE` values, and importantly **drops NAs** (which `padj` often contains for filtered-out OTUs).

#### Your question: Why filter on log2FoldChange again if `lfcThreshold = 1` was already set?

Good catch — this is **somewhat redundant**, but not entirely:
- `lfcThreshold = 1` changes the *p-value calculation* but doesn't itself filter results.
- The `padj < 0.05` filter now means "FDR-significantly above the 1-LFC threshold."
- Adding `abs(log2FoldChange) > 1` is a belt-and-suspenders check. Because of shrinkage and estimation, it's theoretically possible (rare) to have a significant padj with an estimated LFC just under 1. The author wants to guarantee both.

It's defensible but slightly redundant. Many practitioners would omit the second filter.

## 3. Key Concepts

- **R formula syntax (`~`)**: A way to symbolically describe a model. `y ~ x` means "y depends on x." In DESeq2, the response (counts) is implicit, so you only write the right-hand side.
- **Adjusting for covariates**: Including `batch` in the design lets DESeq2 mathematically remove batch effects from the treatment comparison.
- **Negative Binomial model**: A count distribution that handles the overdispersion typical of sequencing data (variance > mean).
- **Multiple testing correction (`padj`)**: With thousands of OTUs, raw p-values yield many false positives. BH correction controls FDR.
- **The `which()` + indexing pattern**: Safer than direct logical indexing when NAs are present, since `res[res$padj < 0.05, ]` would return NA rows, but `res[which(res$padj < 0.05), ]` would not.

## 5. Potential Issues

1. **Design ordering**: As noted, `~ treatment + batch` puts treatment first. Convention is to put the variable of interest *last* (`~ batch + treatment`). Since the code uses explicit `contrast=`, it works either way, but it's slightly nonstandard.
2. **Filtering threshold is arbitrary**: `rowSums >= 10` is lenient. For sparse microbiome data, prevalence-based filtering (e.g., "present in ≥X% of samples") is often more appropriate.
3. **Compositionality not addressed**: Microbiome data is *compositional* (relative, not absolute, abundances). DESeq2 wasn't designed for this; tools like ALDEx2, ANCOM-BC, or corncob are sometimes preferred. This is an active debate in the field — DESeq2 is still widely used and acceptable.
4. **No `lfcShrink()`**: For visualization (e.g., MA plots, ranking), shrunken log fold changes via `lfcShrink()` are usually preferred. They're not needed for the hypothesis test itself, though.
5. **`treatment` factor levels**: This code assumes `treatment` is a factor with levels including exactly `"diseased"` and `"healthy"`. If `treatment` is a character vector or has different labels, this will error.
6. **Redundant LFC filter** (as discussed in Step 5).
7. **No sanity checks**: Production code would verify dimensions, factor levels, and that `colData` rows match `countData` columns.
