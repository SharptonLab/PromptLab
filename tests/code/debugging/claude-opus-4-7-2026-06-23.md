# Debugging Your Bray-Curtis Distance Calculation

## 1. Explain the Error

The error `shapes (50,) and (100,) not aligned: 50 (dim 0) != 100 (dim 0)` means you're trying to perform an operation between two arrays of incompatible lengths. Specifically, `braycurtis` is comparing a vector of length 50 with a vector of length 100, but it requires both inputs to be the same length (since it computes element-wise differences).

## 2. Identify the Cause

The bug is on this line:
```python
distances[i,j] = braycurtis(otu_table[i], otu_table[:,j])
```

Look closely at the indexing:
- `otu_table[i]` → row `i` (a sample), shape `(100,)` ✓ — this is what you want
- `otu_table[:,j]` → column `j` (a taxon across all samples), shape `(50,)` ✗ — this is **wrong**

You accidentally mixed row indexing (`otu_table[i]`) with column indexing (`otu_table[:,j]`). For pairwise sample distances, both arguments should be **rows** (samples), not a row and a column.

## 3. Provide the Fix

```python
import numpy as np
from scipy.spatial.distance import braycurtis, pdist, squareform

# Load OTU table (samples x taxa)
otu_table = np.loadtxt('otu_table.csv', delimiter=',', skiprows=1)
print(f"OTU table shape: {otu_table.shape}")  # (50, 100)

# ---- Option A: Minimal fix to your loop ----
n_samples = otu_table.shape[0]
distances = np.zeros((n_samples, n_samples))

for i in range(n_samples):
    for j in range(n_samples):
        # Both arguments should be ROWS (samples), not columns
        distances[i, j] = braycurtis(otu_table[i], otu_table[j])  # changed otu_table[:,j] → otu_table[j]

# ---- Option B: Faster, idiomatic version ----
# pdist computes pairwise distances in condensed form, squareform converts to full matrix
distances_fast = squareform(pdist(otu_table, metric='braycurtis'))
```

Both produce the same result, but Option B is far faster (vectorized in C) and avoids redundant computation (Bray-Curtis is symmetric and zero on the diagonal).

## 4. Explain Why

- **What was wrong**: `otu_table[:,j]` selects all samples for taxon `j`, giving a length-50 vector. Bray-Curtis dissimilarity is defined as `sum(|x_i - y_i|) / sum(x_i + y_i)` — it requires `x` and `y` to be the same length and represent the same dimensions (taxa abundances). Comparing a sample (length 100) to a taxon profile (length 50) is not mathematically meaningful, even if the shapes happened to match.
- **Why the fix works**: `otu_table[j]` correctly retrieves row `j` (sample `j`'s taxa abundances), so both arguments now live in the same 100-dimensional taxa space.

## 5. Prevention

1. **Mentally label your axes.** For an OTU table with shape `(samples, taxa)`, get in the habit of writing `otu_table[i, :]` and `otu_table[j, :]` explicitly when you want samples. The redundancy makes column-vs-row mistakes more obvious.
2. **Add a shape assertion early in the loop** during development:
   ```python
   assert otu_table[i].shape == otu_table[j].shape
   ```
3. **Prefer library functions over manual loops** when they exist. `scipy.spatial.distance.pdist` is purpose-built for pairwise distances, handles symmetry, and is much harder to misuse.
4. **For microbiome work specifically**, consider `scikit-bio`'s `beta_diversity` function, which takes an OTU table and returns a `DistanceMatrix` object with sample IDs attached — this prevents axis confusion entirely:
   ```python
   from skbio.diversity import beta_diversity
   dm = beta_diversity('braycurtis', otu_table, ids=sample_ids)
   ```
5. **Sanity-check with a known case.** `braycurtis(x, x)` should always be `0`. Running this on a few samples before the full loop catches indexing bugs immediately.

One quick note: depending on whether your OTU table is raw counts or relative abundances, you may also want to normalize rows to sum to 1 before computing Bray-Curtis — though that's a methodology choice, not a bug.
